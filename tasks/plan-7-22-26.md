### 3. AI wrong-recommendation fix — Phase 1 (safety patch)

The exact bug from the screenshot: parent asked about potty training, AI recommended Becca (Pelvic Floor). Root cause: no toddler-development bucket in the specialty taxonomy (four hardcoded arrays/functions in `chat_ai_rag_fixed`), and semantic threshold is deliberately loose (`0.10`) — Becca was the only candidate and the model rationalized the connection via shared "bladder/bathroom" vocabulary.

**Files:**
- `supabase/functions/chat_ai_rag_fixed/index.ts`:
  - New `Toddler Development` category in `detectMessageCategory` (EN/VI/ES keywords), checked *before* Pelvic Floor so potty/toilet/tantrum never fall through.
  - Belt-and-suspenders: hard-clear `matchedExperts = []` when category is `'Toddler Development'` so no candidate even reaches the LLM. Also skips the recurring-topic re-retrieval for this category.
  - Prompt table update: explicit "Potty/toilet training/toddler tantrums/discipline/developmental milestones → No specialist currently on the platform. Do NOT recommend a Pelvic Floor Specialist. Answer with general guidance only." Plus a new disqualifier under the Pelvic Floor row: `✗ NOT for the child's potty/toilet training — Pelvic Floor is the PARENT's own body`.
  - Assistant message metadata now stores `detected_category` alongside `expert_suggestions` (feeds the wrong-match flagging in #4).

**Verified:** typecheck clean, `npm run build` clean, lint clean.

### 5b. CRITICAL — fixed silently-broken semantic expert search

Investigating "why doesn't the AI seem fully aware of expert specialties/descriptions" surfaced a much bigger bug than the taxonomy work above: `find_similar_experts` has referenced a `profiles.expert_embedding` column that **has never existed** on this database (real embeddings live in a separate `expert_embeddings` table). PL/pgSQL doesn't validate column references until execution, so every migration recreating this RPC applied cleanly, but every actual call failed silently and returned `[]` — semantic expert matching has never run in production. The AI chat has relied entirely on the narrower keyword-fallback path this whole time, which is the real reason Shivani was missed originally (her bio would have matched semantically regardless of any keyword list).

**Files:**
- `supabase/migrations/20260722000002_fix_find_similar_experts_broken_embedding_column.sql` — joins `expert_embeddings` on `expert_id` instead of the nonexistent column. All existing eligibility filters unchanged.
- `supabase/functions/generate_expert_embeddings/index.ts` — removed the dead write to the same nonexistent column (was silently failing on every embedding generation, unchecked).
